DATA ANALYSIS

First of all, we need to understand the information described in our CSV. We have a compilation of measurements about the air quality in the city of Barcelona. As we didn’t find any documentation on this dataset, we had to try and find out the meaning of the columns.

Meaning of the Columns

After some time, we ended with the following:

  • nom_cabina: Represents the name of the district inside the city of Barcelona.
  • qualitat_aire: Tells us if the air is good to breathe or not.
  • codi_dtes: Might be just a code related to a specific district.
  • zqa: Is a column from which we cannot extract information, since it only contains 01.
  • codi_eoi: Is also a code related to a specific district.
  • longitud and latitud: Are both data that represent the longitude and latitude of the sample location taken, respectively.
  • hora_o3, hora_o2, and hora_pm10: Represent the time of the day in which the data was taken.
  • qualitat_o3, qualitat_no2, quealitat_pm10: Are qualitative representations of the air quality.
  • valor_o3, valor_no2, valor_pm10: Represent the actual value of the data taken, measured in **µg/m³**.
  • generat: Has information about the date and hour the data was processed.
  • dateTime: Is a numeric code that represent different instances of generat.

Now that we know what every column means, we can start to talk about the data itself. There are **3262 rows** in our Data Set. Apart from the obvious, we can see that there are some instances without actual data, those we’ll need to clean out of our csv.

Location and Quality Values

We can talk about the longitude and latitude values, which tells us that there are several machines extracting the data in different points inside the districts of Barcelona. According to google maps, the instances we checked are **“real”** locations inside the district which they are supposed to be into.

The qualitative values from the qualitat_xx rows are not well defined. As we don’t have documentation, we can’t be sure of the correctness of them. They supposedly correspond to the value of the columns valor_xx, but we don’t know the ranges, in which they can say the quality is **“Buena”** (Good), **“Regular”** (Regular) or **“Pobra”** (Poor), which are the only 3 values of those columns.

The ranges of value_o3, value_n2, and value_pm10 are... (*It is indicated here that the ranges are incomplete in the original text*)

In all 3 columns valor_xx, the data seems to make sense. Even though we don’t know what real data referring to this should look like, we can intuit the difference in measures by the area and the hour it was taken in. For instance, all the measurements taken at dawn seem to be smaller than those taken during the day in the same place. This may have something to do with the differences in traffic during the day. Also the same places seem to have similar measurements throughout the month at the same hours, which tells us that these data might really relate with correct measurements.

RESOURCE NAMING STRATEGY

We will use: **#**

Domain: http://airecalidadbcn.linkeddata.es/

Path: http://airecalidadbcn.linkeddata.es/g05

Ontology:

http://airecalidadbcn.linkeddata.es/g05/ontology#

Resources:

  • **District:** http://airecalidadbcn.linkeddata.es/g05/District/{District_name}
    Description: The name of the district the data comes from.
  • **Quality:** http://airecalidadbcn.linkeddata.es/g05/Quality/{DateTime}

Properties:

Object properties:

  • has_date_time: property of the same district (data = code_dte)
  • has_value: property of the same district (data = code_eoi)
  • has_hash_code: property of the sample (data = longitud, latitud)

Data properties:

  • has_hour_o3: Shows the time when the measuring of o3 was taken.
  • has_value_o3: Shows the amount of o3 contaminant.
  • has_hour_no2: Shows the time when the measuring of no2 was taken.
  • has_value_no2: Shows the amount of no2 contaminant.
  • has_hour_pm10: Shows the time when the measuring of pm10 was taken.
  • has_value_pm10: Shows the amount of pm10 contaminant.
