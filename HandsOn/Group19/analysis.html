Analysis and Design for "Madrid Fluye"

1. Dataset Analysis

The three provided CSV datasets have been analyzed to understand their structure, data types, and content.

Dataset 1: Municipal Public Parking (202625-0-aparcamientos-publicos.csv)

  • Description: Contains information about public parking lots managed by the Madrid City Council.
  • Schema and Data: Each row represents a unique parking lot. The most relevant columns for our project are:
    • PK: A unique numeric identifier for each parking lot. Example: 13468.
    • NOMBRE: The name of the parking lot. Example: "Aparcamiento público. Velázquez - Juan Bravo".
    • LATITUD: Latitude coordinate in decimal format. Example: 40.431752322....
    • LONGITUD: Longitude coordinate in decimal format. Example: -3.706308104....
    • DISTRITO, BARRIO, NOMBRE-VIA: Address information.
  • Quantitative analysis: The dataset contains a list of geolocated entities with static descriptive attributes.

Dataset 2: BiciMAD Stations (bikestationbicimad_csv.csv)

  • Description: Provides near real-time information from the BiciMAD public bicycle service stations.
  • Schema and Data: Each row represents a station. The key columns are:
    • number: The unique identifier number of the station. Example: 2.
    • Name: The descriptive name of the station. Example: "2 - Metro Callao".
    • Address: The postal address of the station. Example: "Calle Miguel Moya nº 1".
    • TotalBases: The total number of docks at the station (capacity). Example: 27.
    • DockBikes: The number of bikes currently docked.
    • FreeBases: The number of free docks currently available.
    • POINT_Y: Latitude coordinate. Example: 40,4204.
    • POINT_X: Longitude coordinate. Example: -3,70569.
  • Quantitative analysis: Similar to the parking lots, it is a list of geolocated entities, but with attributes that can change over time (DockBikes, FreeBases).

Dataset 3: Average Daily Traffic (ADT) on Weekdays (I.M.D. LABORABLES 2024 -2025.csv)

  • Description: It offers aggregated data on traffic intensity in different areas of Madrid, by month and year.
  • Schema and Data: The structure is different; it does not represent individual entities but aggregated measurements.
    • Año and Mes: The measurement period.
    • Interior 1er cinturón, En el 1er cinturón, etc.: Columns representing different geographical areas of the city.
    • Values: The values in the cells represent the average vehicle intensity. Example: 103.019 (the period is a thousand separator).
  • Quantitative analysis: This dataset is a time series of measurements. There are no unique identifiers per row; the key is the combination of Year, Month, and Zone (column). To use it, we must transform its structure from wide (zones in columns) to long (one column for the zone, another for the value).

2. Data Source License Analysis

  • Editor (Publisher): The data comes from two main sources from the Madrid City Council:
    • Madrid City Council's Open Data Portal (datos.madrid.es) for parking and traffic.
    • EMT's Open Data Portal (datos.emtmadrid.es) for BiciMAD stations.
  • Rights Holder: The Madrid City Council and the Municipal Transport Company of Madrid.
  • Original License: As indicated in `datasetRequirements.html`, the license for the City Council's data is the CC0 1.0 Universal Public Domain Dedication. The EMT data is also published under open licenses that allow for its reuse.
  • Proposed License for the Generated Dataset: For the RDF data we will generate, we will propose the Creative Commons Attribution 4.0 International (CC BY 4.0) license. This license is ideal because:
    • It allows for maximum reuse, modification, and distribution.
    • It only requires attribution to the original source, which is good practice and respects the original data publishers.

3. Resource Naming Strategy (URI)

We define a clear and persistent URI strategy, separating the ontology from the data resources.

Base Domain

We will use a fictional but descriptive domain for the project:

http://madridfluye.oeg-upm.net/

Prefixes

  • Ontology (vocabulary): mf:
  • Resources (data): mf-resource:
  • External vocabularies:
    • schema: (for common properties like name, address).
    • geo: (for geographic coordinates).
    • xsd: (for literal data types).

URI Patterns for Resources

  • Parking: We will use the unique identifier from the PK column.
