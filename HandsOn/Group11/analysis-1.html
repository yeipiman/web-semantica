3. Resource Naming Strategy

3.1. URI Design Principles

  • Persistence: URIs must remain stable over time
  • Readability: Readable URIs when possible
  • Semantic opacity: Don't infer resource type from URI alone
  • Consistency: Follow uniform patterns throughout the dataset
  • Simplicity: Avoid unnecessarily complex structures

3.2. Base URI and Namespaces

Base domain http://group11.madrid-data.org/ or http://eventFinder.org/madrid-data/
Ontology namespace @prefix mdo: <http://eventFinder.org/madrid-data/ontology#>
Data namespace @prefix mdr: <http://eventFinder.org/madrid-data/resource/>
Justification Hash (#) for ontology: Small, stable vocabulary - efficient to retrieve together
Slash (/) for data: Large dataset with many instances - individual retrieval needed

3.3. URI Patterns for Resources

Resource Type Pattern Example
Event http://eventFinder.org/madrid-data/resource/event/{id-evento} http://eventFinder.org/madrid-data/resource/event/12546355
Date http://eventFinder.org/madrid-data/resource/date/{YYYYMMDD} http://eventFinder.org/madrid-data/resource/date/20251001

3.4. Identifier Construction Rules

  • Lowercase: All paths in lowercase
  • Hyphens: Separate words with hyphens -
  • No special characters: Avoid spaces, accents, ñ, special symbols
  • Stable identifiers: Use IDs from original dataset when they exist
  • Normalized dates: ISO 8601 format (YYYY-MM-DD, YYYY-MM-DDTHH:MM:SS)
  • No sensitive information: Don't include personal data in public URIs

3.5. Content Negotiation

Data URIs must support content negotiation based on the HTTP Accept header:

Accept Header Response Format Alternative with Extension
text/html Readable HTML page .html
text/turtle RDF Turtle format .ttl
application/rdf+xml RDF/XML format .rdf
application/ld+json JSON-LD format .jsonld
text/n3 N3 format .n3

Implementation options:

  • HTTP 303 redirects: Original URI redirects to format-specific URI
  • Direct serving: Same URI serves different formats based on Accept header
  • Query parameters: Alternative using ?format=ttl

4. Next Steps and Deliverables

4.1. Ontology Development

The next phase involves creating a lightweight ontology that includes:

  • Core classes: Main entities from the datasets
  • Properties: Relationships and attributes with defined domains and ranges
  • Vocabulary reuse: Leverage existing vocabularies (Schema.org, SOSA, Dublin Core)
  • Multilingual labels: rdfs:label and rdfs:comment in English and Spanish
  • Documentation: Clear definitions for all terms

4.2. RDF Data Generation

  • Transform CSV/JSON data to RDF format
  • Apply the defined URI patterns
  • Validate RDF syntax and consistency
  • Generate multiple serializations (Turtle, RDF/XML, JSON-LD)
